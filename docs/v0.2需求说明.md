## 1. 背景

### 1.1 【继承 MVP】已交付能力

[MVP需求说明.md](MVP需求说明.md) 定义的 **stdio + 8 只读业务工具** 已于 2026-06-24 完成 §9 全矩阵验收（A1～A5 / T1～T8 / E1～E8 / C1～C3 / D1～D4）。Agent 可在 Cursor 等本地 MCP Client 中通过自然语言查询 SAP 采购订单、销售订单、库存、发票、主数据等，无需直接拼 OData URL。

### 1.2 【v0.2 新增】为何需要 v0.2

MVP §5.2 明确将以下能力推迟至 v0.2（见 [MVP需求说明.md §5.2](MVP需求说明.md#52-范围外out-of-scope)）：

| MVP 排除项 | v0.2 目标 |
|-----------|----------|
| HTTP / Coze 远程部署 | 启用 Streamable HTTP 传输，供 Coze、远程 Agent 调用 |
| `authenticate` 与 Bearer 鉴权 | HTTP 模式可选 `MCP_REQUIRE_API_KEY` 门禁 |
| `trace_sales_order` 全链路编排 | 修复并验收 SO → 交货 → 开票 → 物料凭证 best-effort 串联 |
| `query_sap_scenario` / `list_sap_scenarios` | 暴露内置 42 端点注册表，支持按 key 动态查询 |
| `get_entity_schema` | 解析 `$metadata`，辅助 Agent 发现 OData 字段 |

v0.2 **不引入** Phase 3 被 SAP Arrangement 阻塞的 7 个工具（v1.0），**不引入** SAP 写操作（v2.0）。

### 6.1 【继承 MVP】通用约定

| 小节 | 内容 |
|------|------|
| §6.1.1 | MCP `tools/call`；stdio **或** HTTP 均返回 `content[].text` JSON |
| §6.1.2 | 成功响应 `results` / `header`+`lines` / `health_check` 结构 |
| §6.1.3 | 错误码表（`SAP_AUTH_FAILED`、`SAP_ARRANGEMENT_REQUIRED` 等） |

**【v0.2 扩展】** Phase 2 工具另增 MCP 层错误码：

| error 码 | 含义 |
|----------|------|
| `MISSING_PARAMS` | 必填入参缺失（如 `salesOrder`、`key`、`service`+`entity`） |
| `MISSING_API_KEY` | `authenticate` 未传 `apiKey` |
| `INVALID_API_KEY` | API Key 与 `MCP_API_KEY` 不匹配 |
| `SCENARIO_NOT_FOUND` | `query_sap_scenario` 的 `key` 不在注册表 |
| `INVALID_SCENARIO` | 场景无可用 path |
| `UNAUTHORIZED` | HTTP 层 Bearer 校验失败（非 MCP tool 响应） |

### 6.3 【v0.2 新增】Phase 2 工具 FR-09～FR-14

#### FR-09 `get_entity_schema`

| 项 | 说明 |
|----|------|
| **用途** | 解析 OData `$metadata`，返回实体属性列表，辅助 Agent 构造 filter / 理解字段 |
| **入参** | `service: string`（如 `API_PRODUCT_SRV` 或完整 `/sap/opu/odata/sap/...`）；`entity: string`（如 `A_Product`） |
| **SAP 依赖** | 对应服务 `$metadata` XML/JSON |
| **成功出参** | `{ service, entity, properties: [{ name, type?, nullable? }], count }` |
| **错误** | 缺参 → `MISSING_PARAMS`；SAP 404 → `SAP_NOT_FOUND` |
| **验收** | `{ "service": "API_PRODUCT_SRV", "entity": "A_Product" }` → `count >= 1` 且含 `Product` 等字段 |

---

#### FR-10 `list_sap_scenarios`

| 项 | 说明 |
|----|------|
| **用途** | 列出内置 SAP 场景/端点注册表（与 `probe-sap-connectivity.js` 同源） |
| **入参** | `{}`（无必填） |
| **成功出参** | `{ count, scenarios: [{ key, category, name, scenario, method, protocol, baseUrl, note? }] }` |
| **SAP 依赖** | 无（纯配置表；可选未来扩展 `SAP_SCENARIO_DIR` 扫描） |
| **规则** | `count >= 33`（当前实现 **42** 项）；每项必有 `key` + `baseUrl` |
| **验收** | 返回含 `sales_order_v4`、`purchase_order_header_v4`、`supplier_invoice_single_v2` 等 key |

---

#### FR-11 `query_sap_scenario`

| 项 | 说明 |
|----|------|
| **用途** | 按场景 `key` 动态执行 OData GET，无需为每个实体单独写工具 |
| **入参** | `key: string`（必填）；`entity?: string`（覆盖 path 中实体段）；`filter?: string`；`top?: number` |
| **成功出参** | `{ key, entity, results: [...] }` |
| **错误** | 未知 key → `SCENARIO_NOT_FOUND`；SAP 401/403 → 标准 SAP 错误码 |
| **规则** | `filter` / `top` 追加为 OData 查询参数；仅 GET |
| **验收** | `{ "key": "sales_order_v4", "top": 1 }` → 200 且 `results` 为数组 |
| **v0.2 不验收** | 对 Phase 3 阻塞 key（如 `purchase_requisition_v4`）的 200 响应 — 允许返回 `SAP_ARRANGEMENT_REQUIRED` |

---

#### FR-12 `trace_sales_order`

| 项 | 说明 |
|----|------|
| **用途** | 按销售订单号 best-effort 串联：销售订单 → 外向交货 → 开票 → 物料凭证 |
| **入参** | `salesOrder: string`（必填） |
| **SAP 依赖** | SAP_COM_0109 + 0106 + 0124 + 0108（探测均为 OK(200)） |
| **成功出参** | `{ salesOrder, legs: [{ step, ok, status, count?, error? }] }` |
| **步骤** | `SalesOrder` → `OutboundDelivery` → `BillingDocument` → `MaterialDocument` |
| **规则** | 任一步 SAP 失败仍返回已收集的 `legs`（第一步 SO 失败则早退）；**不保证**业务上必有下游单据 |
| **验收** | 已知 SO（如 `19`）→ 四步均有 `legs` 条目；SO 步 `ok === true` |
| **修复重点** | MVP 推迟原因「链路字段待验证」— v0.2 须用真实 SO 验证 filter 字段（`ReferenceSDDocument` / `ReferenceDocument`）与计数 |

---

#### FR-13 HTTP 传输

| 项 | 说明 |
|----|------|
| **用途** | 供 Coze、curl、远程 MCP Client 通过 HTTP 调用同一套 13 工具 |
| **启动** | `node mcp-server.js`（无 `--stdio` 且 `MCP_ENABLE_HTTP_TRANSPORT=true`） |
| **端点** | `GET /` → `{ name, version, tools: 20, http: true }`；`/mcp` → Streamable HTTP MCP |
| **会话** | `initialize` 响应头含 `Mcp-Session-Id`；后续请求携带 |
| **与 stdio 关系** | `--stdio` 强制关闭 HTTP；两种模式 **二选一** |
| **验收** | `curl http://127.0.0.1:3000/` 200；`POST /mcp` initialize + tools/list 成功 |

---

#### FR-14 `authenticate`

| 项 | 说明 |
|----|------|
| **用途** | 校验 MCP API Key（Client 预检；与 HTTP Bearer 互补） |
| **入参** | `apiKey: string` |
| **成功出参** | `{ success: true, message: "API key valid" }` |
| **错误** | 缺参 → `MISSING_API_KEY`；不匹配 → `INVALID_API_KEY` |
| **HTTP 门禁** | `MCP_REQUIRE_API_KEY=true` 时，**所有** HTTP 请求须 `Authorization: Bearer <MCP_API_KEY>`，否则 **401** `{ error: "UNAUTHORIZED" }`（在 MCP 层之前拦截） |
| **stdio** | 默认 **不** 强制 API Key（`MCP_REQUIRE_API_KEY=false`） |
| **验收** | 正确 key → success；错误 key → `INVALID_API_KEY`；HTTP 无 Bearer → 401 |

## 11. 风险、假设与依赖

### 11.1 【继承 MVP】假设

SAP 凭证有效、单租户、EPC 接受原始 OData 字段 — 见 [MVP §11.1](MVP需求说明.md#111-假设)。

### 11.2 【v0.2 新增】假设

- `trace_sales_order` 使用的 reference 字段在目标租户数据上可匹配（需用真实 SO 验证）。  
- Coze / 远程 Client 支持 Streamable HTTP MCP 或可通过网关兼容。  
- 公网部署方理解 API Key **不是** SAP 凭证，仍需保护 `user.txt`。

### 11.3 依赖

| 依赖方 | 内容 | v0.2 影响 |
|--------|------|----------|
| SAP Basis | MVP 场景 Arrangement 维持 | 8 工具 + trace 子 API |
| 网络 | SAP + HTTP 客户端可达 MCP 端口 | HTTP 验收 |
| MCP Client | stdio 或 HTTP 协议 | C 类验收 |

### 11.4 风险

| 风险 | 缓解 |
|------|------|
| HTTP 默认绑定 `0.0.0.0` 暴露公网 | 文档强调 NFR-19；默认 example 可改回 127.0.0.1 |
